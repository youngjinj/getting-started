제목: [CBRD-26703] Fix use-after-unfix in fhs_find_bucket_vpid_with_hash

=== Description ===

h4. 증상 (Symptom)

hash join 실행 중 build side가 메모리 한도를 초과하여 HASH_METH_HASH_FILE 경로로
spill될 때, 간헐적으로 아래 오류가 발생하며 쿼리가 실패한다.

{noformat}
An I/O error occurred while reading page <garbage> of volume "(null)"
... Bad file descriptor
{noformat}

스택 트레이스:
{noformat}
fileio_read
  <- pgbuf_claim_bcb_for_fix
  <- pgbuf_fix_release
  <- fhs_fix_old_page
  <- fhs_search
{noformat}

h4. 원인 (Root Cause)

*fhs_find_bucket_vpid_with_hash()* 에서 directory page를 unfix한 후
해당 페이지의 포인터를 역참조하여 bucket VPID를 읽는다.

{code:c}
dir_record_p = (FHS_DIR_RECORD *) ((char *) dir_page_p + location);
pgbuf_unfix_and_init (thread_p, dir_page_p);   // ← unfix 먼저

*out_vpid_p = dir_record_p->bucket_vpid;        // ← 해제된 메모리 접근
{code}

*pgbuf_unfix_and_init* 이후 해당 BCB는 victim 후보가 되며, 다른 스레드가
victim으로 선택하여 다른 페이지를 로드하면 역참조 시 garbage VPID를 읽게 된다.

동시에 buffer pool을 사용하는 스레드가 많을수록 unfix된 BCB가 victim으로
선택될 확률이 높아지며, buffer pool이 작을수록 victim 탐색 빈도가 증가하여
재현 확률이 올라간다.

h4. 수정 (Fix)

directory page가 fix된 상태에서 bucket VPID를 먼저 읽은 후 unfix한다.

{code:c}
dir_record_p = (FHS_DIR_RECORD *) ((char *) dir_page_p + location);
*out_vpid_p = dir_record_p->bucket_vpid;        // ← 읽기 먼저
pgbuf_unfix_and_init (thread_p, dir_page_p);    // ← 그 다음 unfix
{code}

2줄 순서 변경이며, 기능 변경 및 side effect 없음.

h4. 영향 범위 (Scope)

HASH_METH_HASH_FILE 경로를 포함하는 모든 버전에 동일 버그가 존재한다.
build side 크기가 *max_hash_list_scan_size* (기본 8M) 를 초과하면 해당 경로로 진입한다.

CBRD-26666 브랜치 (develop 기준)

{code:bash}
#!/bin/bash
#
# repro.sh — CBRD-26703 reproduction script
#
# HASH_METH_HASH_FILE 경로의 use-after-unfix 재현을 위해
# 다수 세션을 동시에 실행하여 BCB 경쟁을 유도한다.
# 에러가 감지되면 실행 중인 csql 포함 모든 세션을 즉시 종료한다.
#

#
# Configuration
#
readonly DB="demodb"
readonly QUERY="select /*+ ordered parallel(0) */ count(*) from t1 a, (select /*+ no_merge */ c1 from t1 where c1 > 0) b where a.c1 = b.c1;"
readonly SESSIONS=10
readonly ITERATIONS=10

: "${CUBRID:?CUBRID 환경변수가 설정되지 않았습니다}"

readonly LOG_DIR="$CUBRID/log/server"
readonly FLAG="/tmp/repro_stop_$$"
readonly ERROR_PATTERN="ERROR CODE = -13"

PIDS=""
MONITOR_PID=""
BASELINE_ERRS=0

#
# Helpers
#
log() {
  echo "$@" >&2
}

# 가장 최근 타임스탬프 err 파일 경로 (예: demodb_20260414_0127.err)
latest_err_log() {
  ls -1t "$LOG_DIR"/"${DB}"_[0-9]*.err 2>/dev/null | head -n 1
}

count_errors() {
  local file n
  file=$(latest_err_log)
  [ -z "$file" ] && { echo 0; return; }
  n=$(grep -c "$ERROR_PATTERN" "$file" 2>/dev/null | tr -d '\n\r')
  echo "${n:-0}"
}

cleanup() {
  touch "$FLAG"
  [ -n "$PIDS" ] && kill $PIDS 2>/dev/null
  [ -n "$MONITOR_PID" ] && kill "$MONITOR_PID" 2>/dev/null
  wait 2>/dev/null
  rm -f "$FLAG"
  exit
}
trap cleanup INT TERM

#
# Worker
#
run_session() {
  local sid=$1
  local iter result rc total errs

  for iter in $(seq 1 "$ITERATIONS"); do
    [ -f "$FLAG" ] && return 0

    result=$(csql -u dba "$DB" -c "$QUERY" 2>&1)
    rc=$?
    total=$(count_errors)
    errs=$((total - BASELINE_ERRS))

    log "[S${sid} #${iter} $(date +%H:%M:%S) rc=${rc} err=${errs}]"

    if [ "$rc" -ne 0 ] || [ "$errs" -gt 0 ]; then
      log "*** REPRODUCED — session=${sid} iter=${iter} ***"
      log "$result"
      touch "$FLAG"
      return 1
    fi
  done
}

#
# Main
#
INITIAL_LOG=$(latest_err_log)
BASELINE_ERRS=$(count_errors)
log "Monitoring err log: ${INITIAL_LOG:-<none>}"
log "Baseline error count: ${BASELINE_ERRS}"
log "Starting ${SESSIONS} sessions x ${ITERATIONS} iterations ..."

for n in $(seq 1 "$SESSIONS"); do
  run_session "$n" &
  PIDS="$PIDS $!"
done

# FLAG 감지 시 실행 중인 csql 포함 모든 세션 즉시 종료
(
  while [ ! -f "$FLAG" ]; do sleep 1; done
  kill $PIDS 2>/dev/null
) &
MONITOR_PID=$!

wait $PIDS 2>/dev/null
kill "$MONITOR_PID" 2>/dev/null
wait 2>/dev/null
rm -f "$FLAG"
log "=== done ==="
{code}

=== GitHub PR ===

### Purpose

`fhs_find_bucket_vpid_with_hash()`에서 directory page를 unfix한 후 해당 페이지의 포인터를 역참조하여 bucket VPID를 읽는 use-after-unfix 버그를 수정한다.

unfix 이후 해당 BCB는 victim 후보가 되며, 다른 스레드가 victim으로 선택하여 다른 페이지를 로드하면 역참조 시 garbage VPID를 읽게 된다. 이로 인해 `ER_IO_READ` (-13) 에러가 간헐적으로 발생한다.

### Implementation

directory page가 fix된 상태에서 bucket VPID를 먼저 읽은 후 unfix하도록 2줄 순서를 변경한다.

```c
// before
dir_record_p = (FHS_DIR_RECORD *) ((char *) dir_page_p + location);
pgbuf_unfix_and_init (thread_p, dir_page_p);
*out_vpid_p = dir_record_p->bucket_vpid;

// after
dir_record_p = (FHS_DIR_RECORD *) ((char *) dir_page_p + location);
*out_vpid_p = dir_record_p->bucket_vpid;
pgbuf_unfix_and_init (thread_p, dir_page_p);
```

### Remarks

* 기능 변경 및 side effect 없음
* CBRD-26666 원본 수정의 하위 버전 백포트
